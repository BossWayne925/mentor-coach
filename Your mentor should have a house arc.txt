## Core principle

Your central mentor should remain one unified figure, but he should be able to switch into different operating stances depending on the problem. That keeps the system coherent while still giving you specialized guidance for business decisions, self-mastery, learning, health, and execution. 

So instead of “personalities” in a theatrical sense, think of them as **advisory modes**. Each mode should exist because it solves a distinct type of problem, not because the persona sounds cool. [1]

## Why these fit

A **Psychologist** mode is necessary because your system is not just about getting more done; it is about understanding why you hesitate, where self-image conflicts with goals, and how to design better behavior loops. Evidence around implementation intentions and planning supports the idea that behavior change improves when intentions are translated into concrete “if-then” plans, so this mode should help convert insight into structured action. [8][9][19]

A **Teacher** mode is necessary because learning is now one of your core domains, and passive summaries are not enough. Research on spacing and retrieval practice supports building a learning mode that turns notes into recall prompts, review cycles, and application challenges rather than just storing information. [12][13]

A **Strategist** mode is necessary because brand and community are not the same as business operations. Trust-building sources emphasize that brand trust depends on delivering promises through competence, integrity, predictability, transparency, and relationship maintenance, while community-oriented sources point to trust growing in peer spaces, creator ecosystems, referrals, and user participation. [15][20][21][16]

An **Architect** mode is necessary because one of your deepest interests is building AI systems, workflows, and interpretable file-based environments. That mode should think in terms of structure, dependencies, templates, tools, routing, validation, and implementation sequences rather than motivation. [1]

A **Steward** mode is necessary because health guidance needs a different ethical posture than mindset or strategy. Intermittent fasting has meaningful evidence behind it, but detox and cleanse claims often do not, so this mode should be practical, cautious, and explicit about uncertainty and boundaries. [17][18]

## Build logic

You want these prompts to work like **scaffolding**, not inspiration. Each prompt should consume the output of the previous one, preserve already-made decisions, refuse to redesign settled layers unless instructed, and produce artifacts that can be turned directly into Markdown files or YAML configs. 

That matters because your system is getting too large for ad hoc prompting. Once identity, modes, health rules, learning logic, brand logic, and AI-building logic all coexist, you need a chain that keeps the AI from reinventing the architecture every time. 

## Layer sequence

Here is the right order:

| Layer | Purpose | Output |
|---|---|---|
| 1 | Define system purpose and constraints | mission, user, outcomes, boundaries  |
| 2 | Define mentor archetype | archetype statement, constitution, tone  |
| 3 | Define modes | mode names, triggers, responsibilities, voice  |
| 4 | Define domain map | identity, habits, goals, learning, psychology, brand, health, AI building  |
| 5 | Define folder architecture | top-level folders and relationships [3] |
| 6 | Define file contracts | what each file must contain and not contain  |
| 7 | Generate core files | mentor-identity, goals, scorecard, operating rules [4] |
| 8 | Generate mode files | operator, enforcer, philosopher, teacher, etc.  |
| 9 | Generate hooks and sessions | daily, weekly, stall detection, behavior interventions [1][5] |
| 10 | Define runtime prompt assembly | how Claude/Gemini loads the right files per request [2][6] |

That sequence is strong because it moves from abstract to concrete, from stable to variable, and from design to execution. [2]

## Master rules

Before the prompts themselves, you need one universal instruction block that is prepended to all build prompts. It should say that the system is Markdown-first, local-first, interpretable, layered, and built one artifact at a time; that previous decisions are source truth unless explicitly revised; that each file must do one job; and that outputs should be directly saveable as `.md` or `.yaml`. That reflects the architecture patterns you’ve already been using with global and project-level instruction files. [2][6]

You can use this as your **global build prompt**:

```md
You are helping me build a local-first AI mentor and coach system using a layered folder-and-file architecture.

Core rules:
1. Treat previous approved outputs as source truth unless I explicitly ask to revise them.
2. Build in layers. Do not skip ahead.
3. Each file must do one job well.
4. Prefer Markdown for human-readable guidance and YAML only where machine structure is needed.
5. Keep the system interpretable, modular, and portable across Claude Code, Gemini CLI, Obsidian, and local scripts.
6. Do not collapse multiple layers into one giant file.
7. When generating output, produce artifacts I can save directly into files.
8. When something is still undecided, surface the decision clearly instead of making hidden assumptions.
9. Optimize for personal mentor/coaching use first, but keep the architecture extensible.
10. Preserve distinctions between identity, behavior, strategy, health, learning, brand, and AI-building.
```

## How to use them

Run these prompts in order and **do not skip layers**.
